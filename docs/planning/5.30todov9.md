### 🌟 阶段零：共享的底层基础设施改造（基础 MCP Client 引擎）

*这是支撑下面两个方向的先决条件，改动集中在后端框架层。*

**1. 引入标准协议库**

- 在 `pyproject.toml` 中新增官方依赖：`mcp`（Anthropic 提供的 Python SDK）。

**2. 开发 MCP 网关 (`backend/agent/mcp_client.py`)**

- 编写一个 `MCPConnectionManager` 类。
- 实现 `connect(url, transport="sse")`：用于连接内网部署的独立 MCP Server（比如你们公司的财务数据库 Server 或 Jira Server）。
- 实现 `get_available_tools()`：向 Server 发起 `tools/list` 请求，拉取外部系统暴露出来的函数名和 JSON Schema。

**3. 动态工具注册中心 (`backend/agent/tools.py` 重构)**

- 废弃原有的硬编码 `@tool`。
- 编写一个适配器函数：`mcp_tools_to_langchain_tools()`，自动把拉取到的 MCP 工具规范，实时转换为 LangChain / LangGraph 能识别的 `Tool` 对象，并注入到你的 Worker 智能体中。

### 🚀 方向一：数据联邦与 BI 对话式分析 (Conversational Analytics)

**核心目标：** 让 Ragent 能够联合查询本地 GraphRAG 知识、云端业务数据库（通过 MCP）和第三方 CRM（通过 MCP），直接输出高管级别的图文分析报告。

**改造计划安排：**

**Phase 1: 升级 Data Analyst Agent (让它从“本地打工人”变成“跨库指挥官”)**

- **现状**：你的 `data_analyst.py` 只能查本地 MySQL。
- **改动**：将其工具集替换为动态拉取的 MCP 数据库工具（如 `postgresql_mcp_query`, `salesforce_mcp_fetch`）。
- **Prompt 优化**：告诉 Data Analyst：“你现在不仅要写 SQL，还要根据 `salesforce_mcp_fetch` 的 Schema，生成对应的查询参数。当两个数据源的结果返回后，你需要在内存中进行 Join 并得出结论。”

**Phase 2: RRF 多路召回与实时数据的深度融合 (融合层改造)**

- **挑战**：GraphRAG 返回的是非结构化的“社区摘要”或“实体关系”，而 MCP 数据库返回的是结构化的 JSON/CSV。
- **改动**：在 `Synthesize`（合成节点）中引入数据处理引擎（如 `pandas` 或轻量级 JSON 解析）。让大模型先读取 GraphRAG 提供的“业务规则解释”，再读取 MCP 返回的“实时流水”，以此规则去判定流水是否异常。

**Phase 3: 多模态图表生成 (结合你的 v7.0 能力)**

- 既然是 BI 场景，干巴巴的文字是不够的。
- 在生成最终答案前，让大模型调用 Python 工具生成 Echarts 的 JSON 配置项，通过前端 Vue 的 Markdown 渲染器直接画出动态图表。

### 🛸 方向二：复杂跨端业务编排 (Multi-Tool Workflows)

**核心目标：** 让 Ragent 成为自动化工作流引擎。用户一句话，AI 自动跨越 Jira、GitLab、Confluence 和 Slack 完成一条龙操作。

**改造计划安排：**

**Phase 1: 引入“工具检索器” (防 Context Window 爆炸)**

- **挑战**：当企业接入了 20 个系统，产生了 100 多个 MCP Tools 时，把它们的描述全塞给大模型会导致 Token 超载。
- **改动**：复用你的 Milvus 经验。每次拉取到新的 MCP 工具描述后，将其向量化存入 Milvus 的 `tools_collection` 中。用户提问时，先通过语义搜索召回 Top-5 最相关的工具，再喂给 Planner 节点。

**Phase 2: Planner 节点（规划器）的 DAG 重构 (v8.0 深度升级)**

- **现状**：你的 Planner 可能只是生成简单的步骤 1、步骤 2。
- **改动**：引入有向无环图（DAG）规划逻辑。要求 Planner 生成的 JSON 必须包含 `tool_name`、`dependencies`（依赖哪个上游步骤的结果）和 `input_mapping`。
  - *例如*：步骤 2（Slack 发消息）的输入，必须严格引用步骤 1（Jira 查状态）的输出结果。

**Phase 3: 状态继承与副作用管理 (State Management)**

- **改动**：更新 `schemas.py` 的 `GraphState`，增加一个 `tool_outputs` 字典。
- 当调用 GitLab MCP 获得代码 Commit ID 后，必须将其暂存到 `tool_outputs` 中，以便下一个工具（Jira MCP）可以无缝读取。

**Phase 4: 追踪画板体验升级 (Frontend Trace Canvas)**

- **改动**：现在的右侧画板显示的是“本地图谱检索中...”。你需要扩展前端逻辑，当调用外部 MCP 时，画板能打出带有第三方系统 Logo 的实时轨迹（比如：“🔄 正在调用 Jira MCP 获取任务进度...”、“✅ 已通过 Slack MCP 发送通知”）。这种可视化的跨端编排极其震撼。

### 💡 总结与建议优先路径

这两个方向都能让你的项目封神，但难度侧重点不同：

- **方向一（BI 分析）**：偏向“数据处理与融合”，如果你想在面试中强调你对**数据资产化、大模型高维推理**的理解，选这个。
- **方向二（跨端编排）**：偏向“系统工程与自动化”，如果你想强调你对**微服务架构、复杂系统解耦、Agent 自治编排**的掌控力，选这个。
