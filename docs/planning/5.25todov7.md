# Ragent AI “多模态大升级”工程改造计划（ v7.0）迭代规划。 

### Phase 1: 解析层改造 —— 攻克“排版破坏”难题

当前的 L1（1200 字符）/ L2（600 字符）/ L3（300 字符）滑动窗口分块策略虽然对纯文本有效，但会粗暴切断跨页表格或图表的上下文。  

- **引入视觉版面分析 (Layout Analysis)：**

  放弃传统的 `PyPDF2` 或 `pdfplumber`，引入如 **Marker** 或 **Unstructured.io** 这样的文档解析工具。在进行文本提取前，先用目标检测模型（如 YOLO 系列）识别出文档中的“标题”、“段落”、“表格”、“图片”和“页眉/页脚”。

- **多模态对象分离与存储：**

  - **文本：** 继续进入你现有的 L1/L2/L3 层次化 Chunking 流程。  
  - **图片与表格截图：** 独立截取，将二进制文件存入你现有的 MinIO 对象存储中。  
  - **映射关联：** 在 MySQL 的 `parent_chunks` 表中新增字段（如 `associated_media_urls`），在数据库和代码逻辑上使用 Python 建立文本与图表的强关联。

### Phase 2: 抽取层升级 —— 多模态知识入图 (Neo4j)

目前你的项目使用 LLM 对 L2 文本块进行实体/关系抽取，并将 `(Subject, Predicate, Object)` 三元组存入 Neo4j。我们需要将图片和表格信息也“图谱化”。  

- **VLM (视觉大语言模型) 描述生成：**
  - 通过调用类似 Qwen-VL（可通过你现有的 DashScope 平台兼容接入）等视觉模型，对提取出的图片/表格进行深度解读，生成包含关键数据的 Markdown 文本。  
- **多模态节点与边扩展：**
  - 在 Neo4j 中新增节点类型 `ImageNode` 和 `TableNode`。  
  - 利用 VLM 从图表中抽取出实体（例如某财务柱状图中的“2025年营收”、“利润率”），并与原有的文本实体建立 `ILLUSTRATES` 或 `CONTAINS_DATA` 的边，打破文本与图片的语义隔离。

### Phase 3: 检索与融合层优化 —— 跨模态四路召回

你现有的 RRF (Reciprocal Rank Fusion) 融合了 Dense（Qwen text-embedding-v1）、Sparse（BM25）和 Graph 三路召回。我们需要引入第四路：**视觉/多模态语义召回**。  

- **文本到图像检索 (Text-to-Image Retrieval)：** 将 Phase 2 中生成的图表 Markdown 描述文本向量化，存入 Milvus。当用户提问（如“第二季度的营收趋势图说明了什么？”）时，检索系统可以召回相关的图表描述及其 MinIO 下载链接。  

- **RRF 算法公式扩展：** 升级 `rag/utils.py` 中的评分逻辑，加入视觉通道权重。新的数学模型如下：  

  $$RRF_{score} = \frac{w_1}{k + rank_{dense}} + \frac{w_2}{k + rank_{sparse}} + \frac{w_3}{k + rank_{graph}} + \frac{w_4}{k + rank_{visual}}$$

  你可以继续复用之前的网格搜索脚本 `scripts/grid_search_rrf.py` 来重新寻优这四个权重参数。  

### Phase 4: 路由层改造 —— 引入 Multimodal Specialist

现有的 Supervisor 会将意图路由给 RAG Specialist、Local/Global Graph Search 或 Web Searcher 等 6 个 Agent。  

- **新增 Multimodal Specialist (多模态专家 Agent)：**
  - **触发条件：** 当用户的提问中包含“图表”、“曲线”、“图像”等意图，或者检索系统高置信度命中了 `ImageNode` / `TableNode`。
  - **工作流：** 该 Worker 不仅接收检索到的文本片段，还会拉取 MinIO 中的原始图片，将其作为 Prompt 的一部分一并喂给多模态大模型（如 Qwen-VL-Plus）。
  - **Docker 部署更新：** 使用 Docker Compose 确保新的服务组件或本地的 VLM 微服务能与现有的 FastAPI 应用无缝协同。

### 面试高频考点与防御话术

在面试中展示这个模块时，你需要准备好应对以下几个尖锐问题：

1. **“处理含有大量复杂表格的 PDF 时，如何保证大模型的幻觉不会污染数据库？”**
   - *防御策略：* 强调你利用了 **Git 般的版本控制思维**。在 `document_index` 表中，我们将文本和 VLM 解析的表格视为不同的特征分支。如果解析失败，我们可以降级回退到你现有的纯文本提取流程，并且利用现有的 HITL (Human-in-the-Loop) 机制，对于 VLM 提取置信度低的图表强制让人工审核后再入库。  
2. **“多模态检索的延迟如何控制？”**
   - *防御策略：* 提到你在 v6.0 版本中已经实现的 **Semantic Cache**。高频的图表解读查询（例如“帮我总结这篇财报的数据”）可以被以 $\geq 0.95$ 的余弦相似度阈值直接拦截，P99 延迟依然控制在 200ms 级别，无需反复调用耗时的 VLM。  

这个升级计划的工程量比较大，涉及到从解析、数据库存储到检索逻辑的全面重构。
