Ragent AI 引入**行业本体（Ontology）受控图谱提取**的详细改造计划书：

## 核心目标与里程碑

- **业务目标：** 将无约束的自由图谱抽取转化为基于特定行业 Schema 的受控抽取，消除图谱冗余，提升事实一致性。
- **架构目标：** 在现有的 Document Ingestion Flow 中加入本体约束拦截器，并升级评测管线以支持 A/B 对比。
- **产出目标：** 收集改造前后的节点数量、孤岛率、召回率等核心对比实验数据，为学术级别的展示提供坚实的数据支撑。

## 阶段一：领域本体定义与 Schema 层改造

本阶段的重点是建立“图谱法则”，定义系统能够识别和处理的合法数据边界。

- **定义领域字典：** 选取一个具体的垂直测试领域（如医疗或金融），梳理出该领域核心的实体分类（如“疾病”、“症状”、“药物”）和关系类型（如“治疗”、“导致”）。
- **Schema 规范化配置：** 在后端的数据模型层，将上述字典转化为严格的枚举类型结构。所有进入系统的知识，必须能够映射到这些预定义的分类中。
- **图谱数据库约束升级：** 扩展现有的 Neo4j 初始化脚本。为新定义的各类核心实体增加独立的数据约束和索引，确保底层存储严格遵守业务本体结构。

## 阶段二：抽取引擎（Document Ingestion Flow）深度重构

这是整个改造的核心，需要将大模型的“发散思维”收敛为“结构化填空”。

- **Prompt 逻辑升级：** 彻底重写 L2 Chunk 的图谱抽取 Prompt。不再要求模型“提取所有有用的实体”，而是强制模型“在给定的类型列表中进行实体和关系的对齐”。
- **强制结构化输出接入：** 利用大模型的 Structured Output（结构化输出）或 Function Calling 能力，确保模型返回的直接是符合阶段一 Schema 定义的数据格式，杜绝纯文本解析带来的不确定性。
- **合规拦截器（Interceptor）开发：** 在模型输出后、写入图谱数据库前，增加一道硬编码的过滤层。用于丢弃所有不在白名单中的实体类型，并验证关系的逻辑流向是否合法（例如：“药物”不能包含“症状”）。

## 阶段三：数据清洗与知识融合优化

复用并升级现有架构中的 v4.0 数据治理能力，让受控抽取的效果最大化。

- **实体消歧（Entity Resolution）逻辑升级：** 原有的聚类可能只依赖编辑距离。改造后，限定消歧算法只能在“相同类型的实体”之间进行。大幅降低算法复杂度并提高合并的准确度。
- **多源知识对齐准备：** 在实体的元数据中预留外部 ID 字段。为后续整合 MCP 接入的结构化关系型数据库（如 MySQL 里的官方产品表）打下基础。

## 阶段四：评测管线升级与实验数据收集

为了直观展示改造深度，必须产出对比数据图表。

- **对照组与实验组构建：** 在 `scripts` 中建立两套并行的 Ingestion 逻辑，一套保留原生的自由抽取，一套走全新的受控抽取。
- **图谱拓扑指标统计：** 开发一个统计脚本，用于抓取对比提取后的核心数据：总节点下降率、边密度变化、无连接的孤儿节点比例。
- **Ragas 评测管线对比：** 运行现有的自动化评测集，重点对比 `Faithfulness`（事实一致性）和 `Context Precision`（上下文精确度）这两个指标的波动幅度。
- **可视化图表生成：** 扩展 `generate_report.py` 的功能，将上述收集到的图谱瘦身指标和性能提升数据，转化为直观的柱状图和折线图，作为核心汇报素材。
