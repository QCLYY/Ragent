这个自适应混合检索与降级架构的改造方向，不仅工程深度拉满，而且极具落地价值。针对你接下来的技术演进，这份详细的改造计划将严格按照从架构设计到代码落地，再到实验数据收集的闭环来展开。这套组合拳不仅能大幅提升项目的企业级质感，更为你准备技术汇报或学术 PPT 提供了完整的逻辑主线和丰富的数据支撑。

以下是分为四个阶段的详细改造计划：

### 第一阶段：智能化 Query 画像与分级路由 (Intelligent Query Profiling)

当前系统的 `Supervisor` 节点承担了所有路由工作，但在高并发下，每一条 Query 都去调用大模型（如 `qwen-plus`）做意图识别成本太高。我们需要引入一个极轻量级的前置拦截器。  

**1. 改造动作：**

- **构建轻量级意图分类器 (Query Profiler)：** 引入一个极低延迟的分类方案。可以提取真实业务数据，微调一个轻量级本地模型（如 Qwen-1.8B 或 BGE-M3），或者更简单地，使用特征词汇结合 Embedding 相似度匹配来进行分类。
- **定义 L1-L3 分级意图字典：**
  - `L1_FACTUAL` (简单事实/闲聊)：直接路由到 `Direct Answer` 或纯向量检索，跳过多 Agent。  
  - `L2_REASONING` (多跳逻辑推理)：路由到 `Local Graph Search` 与 RAG 专家。  
  - `L3_MACRO_SUMMARY` (宏观全局总结)：路由到 `Global Graph Search`。  

**2. 代码落地建议：** 在 `backend/agent/orchestrator.py` 中，在进入 LangGraph `Supervisor` 之前，增加一个 `query_profiler` 函数。只有判断为复杂查询时，才正式唤醒图谱和多智能体网络。  

### 第二阶段：意图驱动的动态 RRF 权重分配 (Intent-Driven Dynamic RRF)

目前系统的 RRF 权重是静态的环境变量（如 `RRF_WEIGHT_DENSE=0.4`, `RRF_WEIGHT_GRAPH=0.3`）。我们要将其与第一阶段的 Query 画像打通。  

**1. 改造动作：**

- **建立意图-权重映射矩阵 (Weight Matrix)：** 根据 Query Profiler 输出的意图标签，动态下发权重向量 $W = [w_{dense}, w_{sparse}, w_{graph}, w_{visual}]$。
  - 事实类：$W = [0.7, 0.3, 0.0, 0.0]$
  - 推理类：$W = [0.2, 0.1, 0.7, 0.0]$
  - 带多模态意图：$W = [0.2, 0.1, 0.3, 0.4]$
- **改造检索管道：** 在 `backend/rag/utils.py` 中修改 `Three-Channel RRF Fusion` 的实现逻辑，使其能够接收动态传入的权重参数，而不是读取环境变量。  

### 第三阶段：负载感知的自适应降级链路 (Load-Aware Adaptive Degradation)

真实的大厂线上环境随时可能面临流量洪峰。当前的重试机制和异常处理（如 Neo4j 超时降级）偏向单次请求维度的保护，我们需要在系统全局层面增加熔断降级。  

**1. 改造动作：**

- **全局负载监控器：** 利用现有的 Redis，实现一个高频更新的 QPS 计数器和滑动窗口延迟监控。  
- **定义全局系统状态：**
  - `NORMAL` (绿色)：QPS < 50。全量链路放行。
  - `WARNING` (黄色)：QPS 50-100。触发一级降级：在 LangGraph 状态机中设置标志位，**强制跳过 `Critique` 和 `Replan` 事实核查节点**，允许 Draft Answer 直接输出，降低 LLM 交互轮数。  
  - `CRITICAL` (红色)：QPS > 100。触发二级降级：**全局熔断 Neo4j 和 Tavily Web Search**，强制系统退化为纯 Milvus 向量检索和离线缓存。  

**2. 代码落地建议：** 修改 LangGraph 的 `Conditional Edges`。在判断是否进入 `Critique` 节点前，不仅要看前置节点的执行结果，还要拉取 Redis 中的全局 `SYSTEM_STATUS` 标志位。  

### 第四阶段：压测与评测数据收集 (Load Testing & Evaluation)

这部分是为你的技术汇报或 PPT 准备“干货”的最重要环节。

**1. 改造动作：**

- **基准对比自动化：** 扩展目前的 `scripts/run_evaluation.py`，利用 80 个 Golden Dataset，分别在“全量静态链路”和“自适应动态链路”下跑分，对比 Ragas 的四项核心指标和端到端延迟（Latency）。  
- **并发性能压测：** 编写一套 Locust 压测脚本，模拟不同 QPS 的并发流量请求。观察在流量逐渐升高的过程中，系统触发黄色和红色降级状态的具体时间点，以及降级后接口成功率的企稳情况。

**2. 核心输出交付物：**

这一阶段完成后，你将拥有真实环境跑出来的“静态 vs 动态评测对比图”**以及**“并发负载压测熔断曲线图”。这些可视化图表将成为你证明系统改造价值的硬核依据。
