# Ragent AI v4.0 需求规格说明书



### 核心方向一：动态知识库一致性管理与图谱自动治理 (Dynamic Knowledge Consistency & Graph Governance)

目前的系统支持文档切片录入和通过大模型提取图谱，但在真实场景中，文档会更新、作废。构建一套横跨向量库、图数据库和关系型数据库的数据一致性管理体系，是极其硬核的底层能力。  

#### 1. 跨库级联软删除与状态同步 (Cascading Soft-Delete)

**目标：** 当用户删除或更新一份文档时，确保 MySQL、Milvus 和 Neo4j 中的底层知识碎片被彻底且安全地清理，避免出现“幽灵答案”。 **具体操作内容：**  

- **MySQL 状态机改造：** 在现有的 `parent_chunks` 和 `documents` 表 中引入 `is_deleted` 和 `version` 字段，采用软删除（Soft Delete）而非物理删除，保留历史追溯能力。  
- **Milvus 异步清理：** 开发一个后台 Python 服务。当监听到 MySQL 中某文档被标记删除时，根据映射关系提取所有对应的 L3 向量 ID，在 Milvus 中通过 `delete()` 接口执行批量剔除。  
- **Neo4j 边剥离与孤立节点回收：**
  - 编写 Cypher 脚本。由于你已经在 `RELATES_TO` 边上绑定了 `source_chunks` 属性，当 Chunk 失效时，脚本需遍历并移除边数组中的该 Chunk ID。  
  - 如果某条边的 `source_chunks` 变为空，则删除该边。
  - 定期运行 `APOC` (Awesome Procedures on Cypher) 库的清理任务，查找度（Degree）为 0 的孤立实体节点并将其回收。

#### 2. 实体自动消歧与融合 (Entity Resolution & Denoising)

**目标：** 解决不同文档提取出的“同物异名”问题（例如“马斯克”、“Elon Musk”、“特斯拉 CEO”被识别为三个独立节点），控制图谱膨胀。

**具体操作内容：**

- **两阶段同义词检测：**
  - **阶段一（快速召回）：** 利用目前的 Leiden 社区发现算法 锁定局部社区。在社区内部，计算实体名称的编辑距离或轻量级向量相似度。  
  - **阶段二（LLM 确认）：** 将疑似同义的实体节点交由大模型（如你的 Qwen-Plus）进行判断：“节点 A (描述 X) 和节点 B (描述 Y) 是否指代同一客观实体？”。  
- **Cypher 节点融合：** 如果确认为同一实体，使用 Cypher 的 `CALL apoc.refactor.mergeNodes([node1, node2])` 将两者合并，继承所有邻接边，并将关系权重 (Weight) 相加。

#### 3. 时序知识图谱增强 (Temporal GraphRAG)

**目标：** 赋予系统回答带有时间跨度问题的能力，例如“2023年到2025年该产品的架构发生了什么变化？”。

**具体操作内容：**

- **时间戳抽取：** 在文档摄入流程的 L2 切片提取阶段，修改 Prompt，要求 LLM 不仅提取 (Subject, Predicate, Object)，还要尝试提取知识发生的时间边界 (`valid_from`, `valid_to`)。  
- **检索路由增强：** 在 Supervisor 节点 的意图识别中增加“时序敏感度”判断。当触发时序查询时，Local Graph Search 生成的 Cypher 查询需自动带上 `WHERE r.valid_from <= $target_year` 的时间过滤条件。  

### 核心方向二：定量化算法评估框架与 GitOps 自动化流水线 (Quantitative Evaluation & CI/CD Pipeline)

一个高级的算法项目不能仅凭体感说“回答得不错”，必须有实验数据支撑，并具备成熟的代码版本与部署管理能力。这部分能为你产出大量可用于汇报和演示的图表数据。

#### 1. 构建自动化 RAG 评估管道 (Automated Evaluation Pipeline)

**目标：** 引入行业标准的自动化评估指标，定期为系统打分，证明架构优化的有效性。

**具体操作内容：**

- **构建黄金数据集 (Golden Dataset)：** 基于你的测试文档，人工构建 50-100 组 (Question, Ground_Truth_Answer, Source_Context) 作为测试基准。
- **集成 Ragas / TruLens 框架：** 编写一个独立的 Python 评估模块，针对三大核心指标进行量化：
  - *上下文精确度 (Context Precision)：* 检索出的 L3 节点或图谱摘要是否真正包含答案。
  - *答案忠实度 (Faithfulness)：* 最终合成的回答是否出现了幻觉（是否完全基于召回的 Context）。
  - *答案相关性 (Answer Relevance)：* 回答是否直接切中用户的问题。
- **数据可视化：** 将每次评估的得分输出为 JSON 或 CSV，方便后续生成雷达图或折线图，直观展示模型升级或 Prompt 调整带来的性能提升。

#### 2. 三路召回权重的网格搜索与实验设计 (Grid Search for RRF)

**目标：** 通过科学实验找出混合检索的最佳参数，而非拍脑袋决定。

**具体操作内容：**

- 你目前的系统在 Milvus 中使用了 Dense + Sparse + Graph 的三路 RRF 融合。定义其打分公式为：  

  $$RRF\_Score = \frac{w_1}{k+rank_{dense}} + \frac{w_2}{k+rank_{sparse}} + \frac{w_3}{k+rank_{graph}}$$

- **自动化调参脚本：** 编写脚本，在 $[0, 1]$ 范围内以 $0.1$ 为步长，自动遍历 $w_1, w_2, w_3$ 的不同组合。

- **实验执行与记录：** 对于每一种参数组合，运行上述的“黄金数据集”，记录整体评估得分。找出在“泛化问题”和“细节问题”下分别表现最好的参数组合，并将其作为系统的默认配置。

#### 3. GitOps 与容器化 CI/CD 测试集成 (CI/CD Integration)

**目标：** 确保多人协作或复杂图节点逻辑修改时，系统依然稳定，展现极高的工程素养。

**具体操作内容：**

- **GitHub Actions / GitLab CI 流水线：** 编写 `.yaml` 工作流文件。每次推送代码到 `main` 或 `dev` 分支时，自动触发流水线。
- **自动化环境拉起与测试：**
  - 在 CI 环境中基于现有的 `docker-compose.yml` 启动轻量级的 Milvus、Neo4j、Redis 和 MySQL 服务。  
  - 自动运行你 `tests/` 目录下的 `testrag.py` (验证 RAG 管道)、`testapi.py` (验证大模型连通性) 和 `testmilvus.py`。  
  - 测试通过后，自动构建最新的 FastAPI 服务镜像并推送到镜像仓库。
