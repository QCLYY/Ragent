# Ragent AI v3.0 需求规格说明书：GraphRAG 语义网络系统

## 1. 业务背景与升级目标

### 1.1 现状分析

当前 Ragent AI（v2.0）具备优秀的多 Agent 协同和局部碎片化知识检索能力。但在处理**全局性问题**（如：实体关联分析、跨文档趋势总结、复杂多跳推理）时，现有的基准向量检索（Dense + Sparse）容易由于 Top-K 文本块的物理割裂，导致 LLM 无法获得完整的知识关联全景图。  

### 1.2 升级目标

- **非结构化图谱构建**：在文档异步摄取时，利用 LLM 自动提取实体（Entity）、关系（Relation）和三元组，并对实体进行消歧与融合。

- **层次化社区摘要（Community Summarization）**：利用图聚类算法对全量知识网络进行分层聚类，并预先生成社区摘要，具备全局宏观问答能力。

- **混合检索引擎（Hybrid Graph-Vector Search）**：实现“向量检索 + 拓扑图检索 + 社区摘要检索”的三栖融合，大幅提升多跳推理（Multi-hop Reasoning）的准确度。

  ## 2. 核心系统架构与组件变更

  在原有的外围架构基础上，底层存储与索引层引入**图数据库**，与现有的 Milvus、MySQL 形成三维联动。  

  ```
  ┌────────────────────────────────────────────────────────────────────────┐
  │                        RAG Ingestion Pipeline                          │
  │  Document ──► L1/L2/L3 Chunking ──► Vector Indexing (Milvus)          │
  │                    │                                                   │
  │                    └──► LLM Extractor ──► Entity & Relation (Neo4j)   │
  └────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
                         ┌─────────────────────────┐
                         │ Graph Leiden Clustering │ (异步离线计算)
                         └────────────┬────────────┘
                                      │
                                      ▼
                         ┌─────────────────────────┐
                         │  Community Summaries    │ (存储于 MySQL/Redis)
                         └─────────────────────────┘
  ```

  ### 2.1 技术栈新增

  - **图数据库**：Neo4j (Community Edition) 或 NebulaGraph，用于存储实体拓扑网络。
  - **图算法库**：Neo4j Graph Data Science (GDS) 或 NetworkX（Python 库），用于运行 Leiden/Louvain 聚类算法。

  ## 3. 核心功能需求

  ### 3.1 知识图谱自动化抽取管线（Graph Ingestion Pipeline）

  修改原有的 `milvus_writer.py` 逻辑，在对 L3 叶子节点进行向量化的同时，并行触发图谱抽取：  

  - **异步图抽取任务**：利用 Celery 或 FastAPI BackgroundTasks 异步调用大模型，从 L2/L3 文本块中提取出 `(Subject, Predicate, Object)` 三元组。  
  - **实体消歧（Entity Resolution）**：LLM 必须通过 Context 提示词，将“张经理”、“张大伟”、“大伟”等指向同一实体的文本进行合并，避免图节点爆炸。
  - **物理存储关联**：在 Neo4j 中创建节点的物理边（RELATION），同时在边和节点上附带属性：`source_chunk_id`（关联到 MySQL 中的真实文本块）。  

  ### 3.2 离线分层社区聚类（Community Detection & Summarization）

  借鉴 Microsoft GraphRAG 的核心思想，解决宏观全局提问的痛点：

  - **拓扑聚类**：系统每当有大批文档导入后，定时在后台运行 **Leiden 算法**，将整个知识图谱切分为若干个紧密连接的“社区”（Communities）。
  - **分层摘要生成**：
    - **底层社区**：让 LLM 读取该社区内所有的实体、关系和原始 Chunk，生成一份“局部技术报告”。
    - **高层社区**：让 LLM 读取其子社区的摘要，融合成覆盖范围更广的“宏观综述”。
  - **摘要存储**：将生成的各级社区摘要进行文本向量化输入 Milvus，文本实体关系存入 MySQL。  

  ### 3.3 混合图语义检索引擎（Hybrid Graph-Vector Retrieval）

  重构 `rag_utils.py` 中的检索核心，当用户发起提问时，路由支持两种模式：  

  #### 模式一：局部搜索（Local Search - 针对具体实体或多跳推理）

  1. **关键词与向量检索**：先在 Milvus 中检索与问题最相关的 Top-K 文本块。  
  2. **图谱关联外扩**：提取文本块中关联的 Neo4j 实体节点，并在图数据库中向外扩散 1~2 度（1-hop / 2-hop），捞取邻居节点和关系描述。
  3. **上下文重组**：将向量 Chunk + 图三元组文本共同送入 LLM 进行 Rerank 和最终回答。  

  #### 模式二：全局搜索（Global Search - 针对总结性/全局性提问）

  1. **摘要匹配**：直接在 Milvus 的社区摘要索引中进行向量匹配，找出与问题最相关的 N 个社区。
  2. **多级聚合**：并行将这 N 个社区的摘要送入 LLM 进行中间层回答生成，最后由 Supervisor Agent 汇总提炼，杜绝因 Token 溢出导致的上下文丢失。  

  ## 4. 技术方案与算法细节

  ### 4.1 混合得分融合算法 (Enhanced RRF)

  在原有的 Dense + Sparse 双通道 RRF 基础上，将 **Graph 拓扑得分** 引入融合公式。对于候选文本块 $d$，其最终的融合评分为：  

  $$RRF\_Score(d) = \frac{w_1}{k + r_{dense}(d)} + \frac{w_2}{k + r_{sparse}(d)} + \frac{w_3}{k + r_{graph}(d)}$$

  - 其中 $r_{graph}(d)$ 代表该文本块所包含的实体在 Neo4j 图检索中的中心度排名或路径权重。
  - 通过调整 $w_1, w_2, w_3$ 的权重，平衡语义相似度、文本匹配度与知识关联度。  

  ### 4.2 Neo4j 节点与边 Schema 设计

  Cypher

  ```
  // 节点：实体
  CREATE CONSTRAINT FOR (e:Entity) REQUIRE e.id IS UNIQUE;
  // 实体属性：{name: "Qwen-Plus", type: "LLM", description: "..."}
  
  // 边：关系
  // (e1:Entity)-[r:RELATION]->(e2:Entity)
  // 边属性：{description: "由...开发", weight: 0.85, source_chunks: ["chunk_001", "chunk_002"]}
  ```

  ## 5. 非功能性需求与边界防御

  - **图谱防爆炸（Token & Cost Guard）**：图提取非常消耗 Token。系统必须限制单次文档上传的最大图提取深度。对于非核心实体（如低频出现的通用词），设置剪枝策略（Pruning），合并或丢弃度数（Degree）小于 1 的孤立节点。
  - **事务一致性**：当用户在前端删除某份文档时[cite: 1]，删除触发器不仅要清空 MySQL 记录和 Milvus 向量[cite: 1]，还必须在 Neo4j 中执行关联边的级联删除或引用计数减一，防止出现图谱脏数据。













## 🏗️ 第一阶段：基础设施与数据建模

这个阶段的任务是让 Neo4j 跑起来，并定好图谱的“游戏规则”。

- **追加图库节点**：在你的 Docker Compose 堆栈中部署 Neo4j 实例，配置好二进制 Bolt 协议端口（用于后端高效通信）和 HTTP 端口（用于浏览器可视化控制台查看图谱）。  
- **确立唯一性约束**：在 Neo4j 中为“实体节点（Entity）”的名字建立唯一性索引。这是工程上的铁律，防止大模型抽取出重名实体时，图数据库里出现一堆重复的孤立圆圈。
- **设计“骨肉绑定”Schema**：
  - **点（实体）**：存储实体名、类型（如人名、机构、技术栈）和全局描述。
  - **边（关系）**：存储源节点、目标节点、关系类型（如 DEPENDS_ON）。**最关键的要点：边上必须开设一个属性数组，用来存放该关系在 MySQL/Milvus 中对应的原始文本切片 ID（`chunk_id`）**。  

## 🔄 第二阶段：异步注入管线（Data Ingestion）升级

我们需要在你原有的文档切片存储流程中，横向切入一道“图谱提取”工序。  

- **拦截切片流**：当你的系统将文档切入到 L2（中层）或 L3（叶子）级别并准备写入 Milvus 时，同步把这批文本块抄送一份给图谱提取服务。  
- **提示词强约束抽取**：编写一个专门的图提取提示词，利用大模型的结构化输出能力（如 Function Calling 或 JSON Mode），让模型阅读文本并只吐出标准的实体列表和关系列表。
- **图谱事务写入**：
  - 使用“存在则更新，不存在则创建”（Merge）的逻辑写入实体。
  - 创建关系边，并把**当前文本块的 Milvus ID** 动态追加到这条边的 `source_chunks` 数组里。这样，以后你点开图谱上的任意一条线，都能一清二楚地知道它是从哪个具体的合同或文档片段里提炼出来的。  

## 📈 第三阶段：离线图计算（注入全局大局观）

这一步不需要实时触发，通常作为后台的定时任务（Cron Job）或异步批量任务运行。

- **拓扑网络聚类**：编写一个独立脚本，定期把 Neo4j 中的全量图谱结构拉出来，在 Python 内存中运行 **Leiden（莱顿）或 Louvain（鲁汶）社区发现算法**。算法会自动把关系紧密的实体划分到一个“社区”里（例如：把所有和财务相关的实体归为社区 A，IT 相关的归为社区 B），并把社区 ID 写回 Neo4j 节点中。
- **生成社区摘要**：遍历每个社区，把里面所有的实体和关系翻译成白话文丢给大模型，让大模型为这个社区写一份“宏观综述报告”。
- **摘要双向持久化**：
  - 将这份“社区综述文本”存入 MySQL 备查。  
  - 将这份综述文本通过 Embedding 模型向量化，存入 Milvus 的一个全新集合（Collection）中。  

## 🔍 第四阶段：检索引擎（Retrieval）双通道重构

这是最核心的改造。你需要重构你的 RAG 工具层，提供 Local（局部）和 Global（全局）两栖检索能力。  

### 1. 局部检索通道（Local Search - 查具体细节/多跳推理）

- **第一步：向量探路**。先走你现有的 Milvus 混合检索，召回最相关的 Top-K 个原始文本切片 ID。  
- **第二步：图谱外扩**。拿着这几个 ID 去 Neo4j 里逆向查询，捞出所有把这些 ID 挂在边上的关系和实体，并在图谱上向外扩散 1~2 度（捞出它们的邻居）。
- **第三步：上下文合流**。将原始文本切片内容，与图谱里捞出来的三元组事实（如“事实证明：A 控股了 B”）混合拼接，最后通过你原有的 Rerank 引擎过滤后喂给大模型。  

### 2. 全局检索通道（Global Search - 查宏观总结/对比分析）

- **第一步：摘要匹配**。用户的宏观问题（如“公司目前面临哪些主要技术瓶颈？”）不走原始切片库，而是直接去搜你第三阶段建立的 **Milvus 社区摘要库**。  
- **第二步：分批聚合**。捞出关联度最高的几个社区综述报告，由于摘要已经高度浓缩，你可以并行让 LLM 对这几个社区的报告进行中间层解答，最后由主管智能体（Supervisor Agent）融合成最终的深度报告，完美解决爆 Token 和断章取义的问题。

## 🎨 第五阶段：LangGraph 路由编排与前端可视化

最后，把这两套全新的检索武器装备到你的 Agent 军团中。  

- **图工作流升级**：在 LangGraph 的增量流转中，让 Supervisor Agent 增加一层判断：如果用户的问题带有“总结、对比、全景、所有”等字眼，将任务路由给 `Global_Graph_Search` 节点；如果是具体的技术细节或实体查询，路由给 `Local_Graph_Search` 节点。  
- **前端状态机打通**：由于你之前已经搞定了基于 SSE 的流式步骤可视化，现在只需在后端切换检索通道时，向前端 Vue 3 推送新的事件状态（如 `[正在逆向调取 Neo4j 关联图谱...]` 或 `[正在提取 Leiden 社区摘要...]`），并在前端右侧的推理面板中动态高亮展示即可。  

### 🚀 避坑战略提示

在整个流程中，**最容易翻车的是第二阶段（图谱抽取）**。如果让大模型对每个非常碎的 L3 切片都去提图谱，不仅 Token 费用爆炸，而且三元组会极度碎片化。最佳实践是：让大模型读中等粒度的 L2 切片来抽取图谱，但写入 Neo4j 边属性时，绑定它下方包含的所有 L3 叶子节点的 Milvus ID。 这样既保证了语义的连贯性，又实现了精准的向量对齐。
