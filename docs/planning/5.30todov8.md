## Ragent AI 进阶架构升级计划书：自适应推理与图谱自纠错循环

**核心目标：** 将 Ragent AI 从“单向执行的 RAG 流水线”升级为“具备自省能力、能主动规划并按需向 Neo4j 请求补充上下文的自适应智能体系统”。

### 总体里程碑速览

| **阶段**    | **核心任务**                 | **关键产出物**                     | **难度评估** |
| ----------- | ---------------------------- | ---------------------------------- | ------------ |
| **Phase 1** | 状态定义与路由重构           | 支持循环逻辑的 `GraphState` 扩展   | ⭐⭐           |
| **Phase 2** | 前置推理：引入 Planner 节点  | 将复杂查询拆解为多步查询计划       | ⭐⭐⭐          |
| **Phase 3** | 后置反思：引入 Critique 节点 | LLM 驱动的幻觉/忠实度判定器        | ⭐⭐⭐⭐         |
| **Phase 4** | 评测验证与闭环               | 复杂查询准确率（Ragas）的提升报告  | ⭐⭐⭐          |
| **Phase 5** | 前端 Trace Canvas 适配       | 能够展示“打回重做”循环的可视化轨迹 | ⭐⭐           |

### Phase 1：架构设计与图状态扩展 (GraphState)

自适应检索的核心是“状态保持”和“循环”。现有的 LangGraph 状态需要增加用于记录推理过程的字段。

- **修改 `schemas.py` 中的 GraphState：** 增加 `query_plan`（查询计划）、`critique_feedback`（反思反馈意见）、`retry_count`（当前重试次数，防止死循环）和 `is_hallucinated`（幻觉标记）。
- **重构图的收口逻辑：** 目前的终点是 `Synthesize` 节点。新架构下，`Synthesize` 的输出将作为“草稿（Draft）”，传递给下一个节点进行审阅。
- **配置防死循环机制：** 利用你已经实现的 `recursion_limit` 或新增的 `retry_count`，设定最大循环次数（建议设为 2 次）。如果 2 次重试后仍有幻觉，触发降级回复并记录日志。

### Phase 2：开发前置推理模块 (Planner Node)

对于企业级复杂查询，Supervisor 直接分发往往不够，需要一个专门做“任务拆解”的模块。

- **新增 `Planner` 智能体：** 作为一个可选的前置节点。当判定用户问题为复杂/多跳逻辑时，Planner 生成一个基于 JSON 的分步执行计划。
- **更新路由逻辑：** Supervisor 不再只返回单一目标 Agent，而是根据 Planner 的步骤，按顺序或并行触发 `Local Graph Search`、`Web Searcher` 等工作节点。
- **上下文累积：** 确保每个步骤检索到的上下文（Milvus 向量块和 Neo4j 三元组）能够统一追加到 `GraphState.context` 中。

### Phase 3：开发自纠错核心模块 (Critique Node & Conditional Edges)

这是本次升级的“灵魂”，决定了系统的抗幻觉能力。

- **新增 `Critique` 评估节点：** 位于 `Synthesize` 节点之后。输入为“生成的草稿”和“检索到的上下文”。
- **Prompt 工程设计：** 让大模型（推荐使用你配置的 `qwen-max` 等强推理模型）做交叉验证。判定草稿中的每一个事实声明是否都能在上下文中找到直接依据。
- **结构化输出验证：** 强制 Critique 节点输出 JSON，包含 `is_valid` (布尔值) 和 `missing_information` (缺失的关键实体或关系说明)。
- **配置条件路由 (Conditional Edges)：** * 如果 `is_valid == true`：结束图执行，向前端流式输出最终答案。
  - 如果 `is_valid == false` 且重试次数在允许范围内：提取 `missing_information`，构建新的子查询（Sub-query），重新路由至 `Local Graph Search` 或其他 Retriever 进行信息补全。

### Phase 4：离线评测与优化 (Evaluation)

用数据证明你的架构升级是有价值的。

- **利用现有的 Ragas 框架：** 选取你 `golden_dataset.json` 中的 `cross_doc` 和 `global_summary` 这两类复杂问题作为测试集。
- **A/B 测试对比：** 跑一次原版系统（单向流），再跑一次新版系统（带 Critique 循环）。
- **核心观测指标：** 重点对比 `faithfulness`（忠实度：衡量答案是否基于检索内容）和 `answer_relevancy`（回答相关性）。预期忠实度将有显著提升。
- **生成报告：** 导出对比雷达图，这将是你简历或项目演示中最具说服力的“战报”。

### Phase 5：前端与用户体验对齐 (UI/UX)

让用户（或面试官）直观地看到这个高端的“思考与纠错”过程。

- **扩展 SSE 事件流：** 在 `brain.py` 中，为新节点增加事件类型，如 `agent_planning` 和 `self_correction`。
- **升级 Trace Canvas：** 当系统触发重试时，在右侧面板通过动画或不同颜色的节点卡片，清晰地展示出“检测到依据不足 ➔ 重新生成图谱查询 ➔ 补充信息”的完整生命周期。
- **流式输出优化：** 在 Critique 判定通过前，可以向前端输出“正在进行逻辑交叉验证...”的中间态提示，缓解用户等待长链推理时的焦虑。
