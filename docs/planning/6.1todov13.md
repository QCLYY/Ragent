## 项目代号：Ragent 流式增量引擎 (Streaming GraphRAG)

### 核心目标 (The Objective)

将 Ragent AI 现有的“全局停机离线批处理”图谱构建管道 `scripts/run_community_clustering.py`，重构为**高可用、低延迟、增量式**的实时流式摄入系统。实现千万级文档的高并发接入，并在保证数据一致性的前提下，将 LLM 的 Token 成本和全图重算延迟降低 90% 以上。  

### 第一阶段：基础设施重构与消息总线解耦 (Week 1)

**破除单点性能瓶颈，建立大厂标准的分布式流式处理底座。**

- **Task 1: 引入工业级消息队列 (MQ)**
  - **行动：** 在 `docker-compose.yml` 中引入 Kafka 或 Pulsar 容器，彻底替代目前基于 Redis 的 `arq` 轻量级异步队列。  
  - **设计：** 设立三大核心 Topic：`doc_ingest` (文档解析)、`graph_extract` (实体抽取)、`vector_sync` (向量同步)。
- **Task 2: 拆分微服务级消费者组 (Consumer Groups)**
  - **行动：** 将现有的单一 `backend/pipeline/ingestion_worker.py` 拆解为三个独立的无状态 Worker 服务，便于利用 Docker 进行横向扩容（Scale-out）。  
  - **链路梳理：**
    - **Worker-A (Parser):** 消费 `doc_ingest`，执行层级 Chunking 和 PDF 布局分析。  
    - **Worker-B (Extractor):** 消费 `graph_extract`，专门负责高耗时的本体控制 LLM 抽取。此节点配置最大并发数，利用消息队列扛住流量洪峰（削峰填谷）。  
    - **Worker-C (Syncer):** 消费 `vector_sync`，负责 Neo4j 拓扑写入与 Milvus 向量化。  
- **阶段里程碑：** 实现文档上传后立即返回 HTTP 202，后台 Worker 根据 Kafka 堆积量自动分配算力，系统具备极高的抗并发能力。  

### 第二阶段：核心算法演进 —— 增量图聚类 (Week 2)

**这是技术汇报中最硬核的“原理论证”部分，用算法复杂度换取时间效率。**

- **Task 1: 局部拓扑感知与增量补丁 (Local Patching)**
  - **行动：** 当 Worker-C 向 Neo4j 插入新的实体（Entity）与关系（RELATES_TO）时，拦截全局 Leiden 聚类调用。  
  - **算法逻辑：** 获取新节点的邻居（1-hop）所在的原有社区 ID。如果新节点与单一社区 $C_x$ 存在高频/高权重连接，直接执行 `UPDATE` 将其归入 $C_x$，无需启动任何聚类算法。
- **Task 2: 基于模块度的子图重构 (Modularity-based Subgraph Re-clustering)**
  - **行动：** 只有当新文档引发了“跨社区桥接”（例如新节点高频连接了三个不同的历史社区），才触发算法。
  - **算法逻辑：** 提取这三个受影响的社区及其内部节点，构建局部子图。仅对这个局部子图重新运行 Leiden 算法。原有的数万个未受影响的节点保持冻结。  
- **阶段里程碑：** 彻底摒弃全图重算。图谱规模在十万节点级别时，更新延迟从分钟级跌落至秒级。

### 第三阶段：数据一致性与精准刷新 (Week 3)

**解决大模型成本控制（Cost Optimization）这一核心业务指标。**

- **Task 1: 脏位状态机驱动 (Dirty Flag State Machine)**
  - **行动：** 在 MySQL 的 `community_summaries` 表 中新增 `is_dirty` 字段和 `last_modified` 时间戳。  
  - **逻辑：** 增量图聚类算法一旦改变了某社区的成员拓扑，立刻将该社区的 `is_dirty` 标记为 `True`。
- **Task 2: 定向摘要生成与 Milvus 向量更新 (Targeted Summary Generation)**
  - **行动：** 编写一个新的轻量级后台巡检任务 `cron_summary_updater`。
  - **逻辑：** 每隔 N 分钟，扫描 `is_dirty = True` 的社区，仅对这几个社区调用 LLM 重新生成 200-400 字的摘要，并将新摘要的 Embedding 覆盖回 Milvus 中，最后将脏位复位。  
- **阶段里程碑：** 单日新增百篇文档时，全局摘要生成的 Token 消耗骤降 90% 以上。

### 第四阶段：实验设计与数据大表产出 (Week 4)

**为20-30分钟的学术演示报告准备弹药，确保每一项优化都有严谨的数据支撑。**

- **Task 1: 构建算法评测基准 (Benchmarking)**
  - 设计不同规模的图谱注入实验（如 5,000 / 50,000 / 200,000 节点量级）。
  - 记录“原方案（全量 Leiden）” 与“新方案（局部增量）”的时间开销。绘制**时间复杂度对比折线图**。  
- **Task 2: 成本节约量化 (Cost Analysis)**
  - 收集每日增量 100 篇文档场景下的 API Token 账单。
  - 产出“摘要重构 Token 消耗对比数据表”，用柱状图直观展示 90% 级别的大幅削减。
- **Task 3: 提炼 STAR 简历条目 (Resume Polish)**
  - *Situation:* 企业级知识库增量接入导致全图重算极慢、大模型 Token 成本高昂。
  - *Task:* 负责重构图谱摄入流水线，设计流式并发处理与低延迟增量更新架构。
  - *Action:* 引入 Kafka 解耦三大计算节点；参考前沿图聚类论文，研发并落地局部子图增量补丁算法以替代全局 Leiden；设计脏位驱动的精准局部摘要重构机制。  
  - *Result:* 在十万级节点规模下，系统更新延迟缩短百倍（20分钟降至秒级）；大模型重构 Token 成本节省 95% 以上，支撑百万级并发文档的流式摄入。
