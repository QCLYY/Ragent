针对“动态数据管道与增量更新”以及“DevOps 容器化”，我为你梳理了一份详细的改造计划。这份计划不仅是一份开发指南，更是你后续在面试中向面试官展示**系统架构演进思维**的剧本。

我们将整个改造分为四个阶段，从底层数据结构设计到最终的工程化部署。

### 第一阶段：底层状态追踪与架构设计 (Data Modeling & State Tracking)

要实现增量更新，系统必须具备“记忆力”，知道哪些数据是旧的，哪些是新的。

- **1. 建立文档指纹机制 (Document Fingerprinting)：**
  - 在 MySQL 中扩展 `document_index` 表，增加 `file_hash` (如 SHA-256) 和 `last_modified` 字段。
  - 每次上传文档时，先计算 Hash。如果 Hash 一致，直接跳过处理；如果 Hash 改变，标记为“待更新”状态。
- **2. 建立血缘关系映射 (Data Lineage)：**
  - 目前系统有 L1/L2/L3 的层级 Chunk。需要确保每个 Chunk（在 MySQL 中）、每个 Vector（在 Milvus 中）和每个 Entity/Relation（在 Neo4j 中）都**强绑定**一个 `source_document_id`。
  - 这是局部删除和局部更新的基石。

### 第二阶段：核心增量更新逻辑开发 (Incremental Sync Core)

这是最具技术含量的部分，主要使用 Python 编写对三种不同数据库的精准控制逻辑。

- **1. 向量库 (Milvus) 的精准剔除：**
  - 当文档更新时，不要清空整个 Collection。先根据 `source_document_id` 查询出所有相关的旧 L3 Chunk 的 Vector IDs。
  - 调用 Milvus 的批量删除 API 删掉这些旧向量，然后再插入新生成的向量。
- **2. 图数据库 (Neo4j) 的局部重建与孤儿回收：**
  - **删除旧边：** 编写 Cypher 语句，找到带有旧 `source_document_id` 属性的关系边（Edges）并删除。
  - **孤儿节点清理 (Garbage Collection)：** 边被删除后，某些实体节点（Nodes）可能不再与任何其他节点连接。编写一个定时执行的 Cypher 清理脚本：“匹配所有度数（Degree）为 0 的节点并删除”。
  - **安全插入：** 依然使用 `MERGE` 语句插入新提取的图谱三元组，确保不会产生重复实体。
- **3. 避免全局计算浪涌：**
  - 限制 Leiden 社区聚类算法的触发频率。不要求每次文档更新都立刻重新聚类，而是设定一个阈值（例如：增量节点超过图谱总节点的 5% 时），在夜间低峰期触发一次全量聚类更新。

### 第三阶段：异步任务队列与解耦 (Asynchronous Task Queue)

如果把增量更新的逻辑全塞在 FastAPI 的请求处理函数里，前端会因为超时而崩溃。

- **1. 引入轻量级消息队列：**
  - 利用系统现有的 Redis，结合 Python 的 `Celery` 或 `RQ (Redis Queue)` 库。
  - FastAPI 接收到文档变更请求后，立刻返回 “Task Accepted (HTTP 202)”，将计算密集型的任务（计算 Hash、大模型提取、更新 Milvus/Neo4j）推入 Redis 队列。
- **2. Worker 进程分离：**
  - 编写一个独立的 Python Worker 脚本，专门从 Redis 取出任务并在后台默默执行数据库的增量更新。

### 第四阶段：容器化编排与 DevOps 规范 (Containerization & DevOps)

向企业级看齐，展现你的工程素养。

- **1. 完善 Docker 编排 (Docker Compose)：**
  - 将你新写的 Worker 进程也写进 `docker-compose.yml` 中。
  - 配置合理的依赖关系（`depends_on`），确保 MySQL、Redis、Milvus、Neo4j 先启动，然后才是 FastAPI Web 节点和后台 Worker 节点。
  - 设置资源限制（如限制大模型提取 Worker 的最大内存占用，防止 OOM 拖垮整个宿主机）。
- **2. 规范化代码流 (Git Workflow)：**
  - 将整个增量更新的改造作为一个独立的 Feature 分支进行开发。在代码提交时，写清晰的 Commit Message（如 `feat(pipeline): implement async neo4j incremental merge`）。
