# Ragent AI v2.0 需求规格说明书：多智能体协同与人类协同（HITL）系统

## 1. 业务背景与升级目标

### 1.1 现状分析

当前 **Ragent AI (v1.0)** 已实现了基于 LangGraph 的单兵 RAG 流水线，具备混合检索、三级层次化切片（L1/L2/L3）及流式响应能力。然而，单一 RAG 专家面对需要实时联网搜索或复杂结构化数据（如 SQL 查询）的复合型任务时，容易因知识库边界限制而产生幻觉或拒绝回答。  

### 1.2 升级目标

- **多 Agent 协同**：将单体 RAG 架构升级为分布式多智能体网络，引入主管智能体（Supervisor）进行任务拆解与分发。
- **人类协同（Human-in-the-loop, HITL）**：引入显式中断机制，在置信度低或高风险操作时挂起工作流，允许人工介入干预。
- **全栈监控与可控性**：前端 UI 从传统的单体流式聊天升级为动态多 Agent 协同看板，支持可视化状态回溯。

## 2. 核心系统架构与 Agent 拓扑

系统采用 **Supervisor-Workers（主管-员工）** 模型，所有状态流转均由 LangGraph 状态机进行强控制。  

### 2.1 架构拓扑图

### 2.2 智能体角色定义

| **Agent 名称**                  | **核心职责**                                                 | **依赖组件 / 工具**                       |
| ------------------------------- | ------------------------------------------------------------ | ----------------------------------------- |
| **Supervisor Agent (主管)**     | 负责解析用户初始意图，将复杂任务拆解，并动态路由给下方各 Specialist Worker；汇总各 Worker 的执行结果并输出最终回答。 | 大语言模型（Qwen-Plus）、条件路由节点。   |
| **RAG Specialist (知识库专家)** | 处理涉及私有文档的深度查询。保留现有的三级切片、混合检索、文档评分（Grading）与重写（Rewrite）机制。 | Milvus 2.5、MySQL、DashScope Rerank API。 |
| **Web Searcher (联网搜索专家)** | 处理涉及实时新闻、时效性信息或私有知识库未覆盖的通用常识查询。 | Tavily API 或 Serper API。                |
| **Data Analyst (数据分析师)**   | 处理涉及结构化数据统计、报表分析的查询。自动生成并运行 SQL。 | MySQL 数据库（只读权限连接）。            |

## 3. 核心功能需求

### 3.1 多 Agent 路由与动态决策

- **意图识别与拆解**：Supervisor 必须支持**单路由**（直接分发给某个 Worker）与**并行拆解**（如：同时调用 RAG 专家获取内部政策，调用联网搜索获取行业最新动态）。
- **上下文聚合**：当多个 Worker 并行结束时，Supervisor 需将各 Worker 返回的 `messages` 写入 LangGraph 全局 State，进行重组后统一生成最终文本。

### 3.2 人类协同（HITL）中断与恢复机制

系统必须支持以下两种中断触发场景：

> 场景 A：低置信度防御（RAG Specialist 触发）
>
> - **触发条件**：当 RAG Specialist 在执行 `grade_documents` 节点时，若连续 **2** 次重写 Query 后的相关性得分均低于设定阈值（例如 **0.4**）。  
> - **系统行为**：LangGraph 触发 `interrupt()`，图状态挂起，向前端发送特定的 SSE 中断事件流。  
> - **人工干预**：系统在前端弹窗提示管理员，管理员可执行：
>   1. **强行补充**参考文本（注入临时上下文）；
>   2. **修改**用户的原始 Query 并点击“重试”；
>   3. 点击“终止”，由 LLM 直接基于现有模糊信息回答或致歉。

> 场景 B：高风险操作风控（Data Analyst 触发）
>
> - **触发条件**：Data Analyst 生成了涉及跨表联查、敏感字段读取或写入尝试（非 SELECT 语句）的 SQL。
> - **系统行为**：图工作流挂起，不直接执行数据库调用。
> - **人工干预**：前端展示即将执行的 SQL 语句，由管理员点击 **“批准执行”** 或 **“驳回修改”**。

### 3.3 前端 UI/UX 全面升级 (Vue 3 SPA)



为了配合后端的多 Agent 架构，前端界面必须进行重构：

- **多 Agent 协同流式看板（Trace Canvas）**：
  - 取消原本单一的聊天气泡，右侧引入折叠式“推理追踪面板”。  
  - 利用现有的 SSE 流，动态高亮显示当前处于激活状态的 Agent（例如：`Supervisor [思考中...]` $\rightarrow$ `Web Searcher [联网搜索中...]`）。  
- **审批交互界面（HITL Modal）**：
  - 当监听到后端的中断事件时，聊天输入框和流式输出处于锁定状态。  
  - 屏幕中央弹出“人类协同干预面板”，清晰展示：当前挂起的 Agent 节点、触发中断的原因、当前全局 State 的快照。

## 4. 技术方案与数据架构变更

### 4.1 状态机设计 (LangGraph State)

全局 State 不再仅仅包含 `messages` 列表，必须扩展为支持多任务同步的数据结构：  

Python

```
from typing import Annotated, Sequence, TypedDict
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class GlobalAgentState(TypedDict):
    # 现有的聊天历史消息
    messages: Annotated[Sequence[BaseMessage], add_messages]
    # 当前激活的 Worker 子任务队列
    next_worker: str
    # Worker 各自汇报的数据片段
    worker_outputs: dict
    # 人类介入时的输入指令或修正文本
    human_interfered_input: str
```

### 4.2 持久化层变更 (MySQL + Redis)



- **MySQL 状态检查点 (Checkpointer)**：基于你现有 MySQL 架构扩展实现 `MySQLSaver`。每一轮 Worker 的流转状态必须序列化并持久化进 MySQL 表 `agent_graph_checkpoints` 中，以便支持长达数小时甚至数天的“中断挂起”与“状态回溯”。  
- **Redis 锁与热缓存**：利用 Redis 对挂起状态的 Session 设置分布式锁，防止用户在 Agent 处于人工中断审批等待期间，连续发送新消息导致状态机死锁。  

### 4.3 向量检索流增强 (Milvus)

[cite: 1]

- **多 Agent 数据隔离（标量过滤）**：在 Milvus 集合中增加标量字段 `source_agent_tags`[cite: 1]。在未来的多 Agent 扩展中，允许不同的 Worker 根据自身权限，在向量检索时动态注入 `expr="source_agent_tags in ['public', 'it_dept']"` 进行硬隔离过滤。

## 5. 非功能性需求与边界防御

  

- **死循环检测（Dead-loop Detection）**：在 LangGraph 中必须配置 `max_concurrency`（最大流转步数，建议设为 **15-20**）[cite: 1]。防止 Supervisor 与 Workers 之间由于意图解析不清，产生无限互相调用的恶性循环。
- **断流与优雅降级**：
  - 若联网搜索 API 出现欠费、频控或超时，系统需自动触发 fallback，将任务降级流转至 `RAG Specialist` 利用本地冷数据回答[cite: 1]。
  - 保留前端的 `AbortController` 功能，允许用户在任意 Agent 节点协同期间手动掐断整个图的运行[cite: 1]。
